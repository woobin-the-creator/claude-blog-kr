목표를 향해 Claude를 계속 일하게 하기: /goal 명령어

2026년 6월 25일 · 읽는 데 약 6분 · 카테고리: Claude Code 원문: Keep Claude working toward a goal (Claude Code 공식 문서 · 한글 번역본)
참고: /goal은 Claude Code v2.1.139 이상에서 동작합니다.

/goal 명령어는 완료 조건(completion condition)을 설정해 두고, 매 단계마다 사람이 다시 지시하지 않아도 Claude가 그 조건을 향해 계속 일하도록 만듭니다. 한 턴(turn)이 끝날 때마다 작고 빠른 모델이 조건이 충족됐는지 확인합니다. 충족되지 않았다면 Claude는 제어권을 사용자에게 돌려주는 대신 다음 턴을 시작합니다. 조건이 충족되면 목표는 자동으로 해제됩니다.

검증 가능한 종료 상태가 있는, 비중 있는 작업에 목표를 사용하세요.

  • 모든 호출 지점이 컴파일되고 테스트가 통과할 때까지 모듈을 새 API로 마이그레이션하기
  • 모든 인수 기준(acceptance criteria)이 충족될 때까지 설계 문서를 구현하기
  • 각 파일이 정해진 크기 예산 아래로 떨어질 때까지 큰 파일을 집중된 모듈들로 쪼개기
  • 큐가 빌 때까지 라벨이 달린 이슈 백로그를 처리하기

이 글에서 다루는 내용:

  • 세션을 계속 돌리는 방법 비교: /loop, Stop 훅, 자동 모드(auto mode)
  • 목표 설정하기와 효과적인 조건 쓰기
  • 상태 확인하기, 조기 해제하기, 비대화형으로 실행하기
  • 평가가 어떻게 동작하는지와 요구 사항

세션을 계속 돌리는 방법 비교

현재 세션을 프롬프트 사이에도 계속 돌리는 방법은 세 가지입니다. 다음 턴을 무엇이 시작시키는지를 기준으로 고르세요.

방식다음 턴이 시작되는 시점멈추는 시점
/goal 이전 턴이 끝났을 때 모델이 조건 충족을 확인했을 때
/loop 일정 시간 간격이 지났을 때 사용자가 멈추거나, Claude가 작업이 끝났다고 판단했을 때
Stop 훅 이전 턴이 끝났을 때 사용자의 스크립트나 프롬프트가 결정할 때

/goal과 Stop 훅은 둘 다 매 턴이 끝난 뒤에 발동합니다. /goal은 세션 범위(session-scoped)의 단축 기능입니다. 조건을 입력하면 현재 세션에서만 활성화됩니다. 반면 Stop 훅은 설정 파일에 저장되어 그 범위의 모든 세션에 적용되며, 결정론적 확인을 위해 스크립트를 실행하거나 모델이 평가하도록 프롬프트를 실행할 수 있습니다.

자동 모드(auto mode)는 그 자체로는 한 턴 안에서 도구 호출을 자동 승인할 뿐, 새 턴을 시작시키지는 않습니다. Claude는 작업이 끝났다고 판단하면 멈춥니다. /goal은 매 턴이 끝난 뒤 사용자의 조건을 확인하는 별도의 평가자(evaluator)를 더하므로, 완료 여부를 작업을 수행한 모델이 아니라 새로운(fresh) 모델이 결정합니다. 둘은 상호 보완적입니다. 자동 모드는 도구별 확인을 없애고, /goal은 턴별 확인을 없앱니다.

팁: 위 방법들은 현재 세션을 계속 돌립니다. 야간 테스트나 아침 분류 작업처럼, 열려 있는 세션과 무관하게 독립적으로 실행되는 작업을 예약할 수도 있습니다. 클라우드 루틴과 데스크톱 예약 작업에 대해서는 예약 옵션 비교를 참고하세요.

/goal 사용하기

세션당 활성화할 수 있는 목표는 하나입니다. 같은 명령어가 인자(argument)에 따라 목표를 설정하거나, 확인하거나, 해제합니다.

목표 설정하기

/goal 뒤에 충족하고 싶은 조건을 붙여 실행하세요. 이미 활성화된 목표가 있으면 새 목표가 그것을 대체합니다.

/goal test/auth의 모든 테스트가 통과하고 lint 단계가 깨끗하다

목표를 설정하면 곧바로 한 턴이 시작되며, 조건 자체가 지시문이 됩니다. 별도의 프롬프트를 보낼 필요가 없습니다. 목표가 활성화된 동안에는 ◎ /goal active 표시가 나타나 목표가 얼마나 오래 실행 중인지 보여줍니다.

매 턴이 끝나면 평가자는 조건이 충족됐는지 또는 충족되지 않았는지를 설명하는 짧은 이유를 반환합니다. 가장 최근 이유는 상태 화면과 트랜스크립트(transcript)에 표시되어, 다음으로 Claude가 무엇을 향해 일하는지 볼 수 있습니다.

참고: 목표는 조건이 충족되거나 사용자가 /goal clear를 실행할 때까지 계속 돌아갑니다. 인자 없이 /goal을 실행하면 지금까지 소모한 턴 수와 토큰을 볼 수 있습니다.

효과적인 조건 쓰기

평가자는 Claude가 대화에 드러낸 내용을 기준으로 조건을 판단합니다. 평가자는 스스로 명령어를 실행하거나 파일을 읽지 않으므로, 조건은 Claude 자신의 출력으로 증명할 수 있는 것으로 작성해야 합니다. "test/auth의 모든 테스트가 통과한다"가 효과적인 이유는, Claude가 테스트를 실행하고 그 결과가 평가자가 읽을 수 있도록 트랜스크립트에 남기 때문입니다.

여러 턴에 걸쳐 잘 버티는 조건은 보통 다음을 갖추고 있습니다.

  • 하나의 측정 가능한 종료 상태: 테스트 결과, 빌드 종료 코드, 파일 개수, 빈 큐 등
  • 명시된 확인 방법: Claude가 그것을 어떻게 증명해야 하는지, 예를 들어 "npm test가 0으로 종료한다" 또는 "git status가 깨끗하다"
  • 지켜야 할 제약: 그 과정에서 바뀌면 안 되는 것, 예를 들어 "다른 테스트 파일은 수정하지 않는다"

조건은 최대 4,000자까지 가능합니다.

목표가 얼마나 오래 실행될지 한계를 두려면 조건에 턴 수나 시간 절(clause)을 포함하세요. 예: 또는 20턴 후에 멈춘다. Claude는 매 턴 그 절에 대한 진행 상황을 보고하고, 평가자는 대화 내용으로 그것을 판단합니다.

상태 확인하기

인자 없이 /goal을 실행하면 현재 상태를 볼 수 있습니다.

/goal

목표가 활성화돼 있으면 상태에 다음이 표시됩니다.

  • 조건
  • 얼마나 오래 실행됐는지
  • 몇 개의 턴이 평가됐는지
  • 현재 토큰 사용량
  • 평가자의 가장 최근 이유

활성화된 목표는 없지만 세션 초반에 달성된 목표가 있었다면, 상태에 달성된 조건과 함께 그 소요 시간, 턴 수, 토큰 사용량이 표시됩니다.

목표 해제하기

/goal clear를 실행하면 조건이 충족되기 전에 활성 목표를 제거합니다.

/goal clear

stop, off, reset, none, cancel은 clear의 별칭(alias)으로 인식됩니다. 새 대화를 시작하려고 /clear를 실행해도 활성 목표가 제거됩니다.

활성 목표가 있는 상태로 재개하기

세션이 종료될 때까지 활성화돼 있던 목표는, --resume 또는 --continue로 그 세션을 재개하면 복원됩니다. 조건은 그대로 이어지지만, 턴 수·타이머·토큰 사용량 기준점은 재개 시 모두 초기화됩니다. 이미 달성됐거나 해제된 목표는 복원되지 않습니다.

비대화형으로 실행하기

/goal은 비대화형 모드(non-interactive mode), 데스크톱 앱, 그리고 원격 제어(Remote Control)에서 동작합니다. -p로 목표를 설정하면 한 번의 호출로 루프 전체를 완료까지 실행합니다.

claude -p "/goal CHANGELOG.md에 이번 주에 머지된 모든 PR 항목이 들어 있다"

비대화형 목표를 조건 충족 전에 멈추려면 Ctrl+C로 프로세스를 중단하세요.

평가가 어떻게 동작하는가

/goal은 세션 범위의 프롬프트 기반 Stop 훅을 감싼 래퍼(wrapper)입니다. Claude가 한 턴을 마칠 때마다 조건과 지금까지의 대화가, 설정된 작고 빠른 모델(기본값은 Haiku)에 전달됩니다. 그 모델은 예/아니오 결정과 짧은 이유를 반환합니다. "아니오"는 Claude에게 계속 일하라고 알리며, 그 이유를 다음 턴을 위한 안내로 함께 전달합니다. "예"는 목표를 해제하고 트랜스크립트에 달성 항목을 기록합니다.

평가자는 사용자의 세션이 설정된 제공자(provider)에서 실행됩니다. 평가자는 도구를 호출하지 않으므로, Claude가 이미 대화에 드러낸 것만 판단할 수 있습니다.

참고: 평가에 드는 토큰은 사용자의 제공자에 설정된 작고 빠른 모델로 과금되며, 보통 메인 턴 사용량에 비하면 무시할 만한 수준입니다.

요구 사항

/goal은 신뢰(trust) 대화를 수락한 워크스페이스에서만 동작합니다. 평가자가 훅 시스템의 일부이기 때문입니다. 또한 disableAllHooks가 어느 설정 레벨에서든 켜져 있거나, 관리형 설정에서 allowManagedHooksOnly가 켜져 있으면 /goal을 쓸 수 없습니다. 두 경우 모두 명령어가 아무 일도 하지 않고 조용히 넘어가는 대신, 그 이유를 알려 줍니다.

함께 보기

  • /loop로 프롬프트 반복 실행하기: 조건이 충족될 때까지가 아니라, 시간 간격마다 다시 실행
  • 프롬프트 기반 훅: 맞춤 평가 로직이 필요할 때 직접 Stop 훅 작성하기
  • 자동 모드: 도구 호출을 자동 승인해 각 목표 턴이 사람 개입 없이 돌아가게 하기
  • 예약 비교: 열려 있는 세션과 무관하게 일정에 맞춰 작업 실행하기
