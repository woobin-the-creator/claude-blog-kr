Claude Code 길들이기: CLAUDE.md 파일, 스킬, 훅, 룰, 서브에이전트 그 외

2026년 6월 18일 · 읽는 데 약 5분 · 카테고리: Claude Code 원문: Steering Claude Code: CLAUDE.md files, skills, hooks, rules, subagents and more (한글 번역본)

개요

Claude Code는 Claude의 동작을 맞춤 설정하고 지시하기 위한 7가지 방법을 제공합니다. CLAUDE.md 파일, 룰(rules), 스킬(skills), 서브에이전트(subagents), 훅(hooks), 출력 스타일(output styles), 그리고 시스템 프롬프트 덧붙이기입니다. 각 방법은 지시가 언제 로드되는지, 세션 동안 어떻게 유지되는지, 컨텍스트 비용이 얼마인지에서 차이가 납니다.

7가지 방법

CLAUDE.md 파일

CLAUDE.md 마크다운 파일은 세션 시작 시 로드되어 세션 내내 유지됩니다. 빌드 명령어, 디렉터리 구조, 모노레포 구조, 코딩 규칙, 팀 관례를 담기에 가장 적합합니다.

두 가지 종류가 있습니다.

  • 항상 로드되는 유형: 루트 CLAUDE.md 파일은 세션 시작 시 로드되며, 컨텍스트 압축(compaction) 이후에 다시 읽힙니다.
  • 필요할 때 로드되는 유형: 하위 디렉터리의 CLAUDE.md 파일은 Claude가 해당 디렉터리의 파일에 접근할 때 로드됩니다.
모범 사례: CLAUDE.md는 200줄 이내로, 담당자를 한 명으로 유지하세요. 모노레포에서는 하위 디렉터리별 CLAUDE.md를 만들어 각 팀이 자기 팀의 규칙만 로드하도록 하세요. 모든 팀이 지시를 덧붙이다 보면 파일이 비대해지고, 관련 여부와 무관하게 모든 세션에서 토큰을 소모하게 됩니다.

룰(Rules)

룰은 .claude/rules/에 저장되는 마크다운 파일로, 제약 조건이나 규칙을 명시합니다. 범위가 지정되지 않은 룰은 세션 시작 시 로드되고 압축 시 다시 주입됩니다. 경로 범위가 지정된 룰은 관련 파일을 건드릴 때만 로드되어 토큰 낭비를 줄입니다.

경로 범위 룰 예시:

---
paths:
  - "src/api/**"
  - "**/*.handler.ts"
---
모든 API 핸들러는 처리 전에 Zod로 입력을 검증해야 한다.
모범 사례: 여러 위치의 파일에 걸쳐 영향을 주지만 조직 전체에 적용할 필요는 없는 횡단 관심사에는 경로 범위 룰을 사용하세요.

스킬(Skills)

스킬은 .claude/skills/ 아래에 폴더 형태로 존재하며, 지시·스크립트·리소스를 담습니다. 세션 시작 시에는 스킬 이름과 설명만 로드되고, 슬래시 명령어나 자동 매칭으로 호출될 때 본문 전체가 로드됩니다.

예로 /code-review, 배포 워크플로, 릴리스 체크리스트 등이 있습니다. 압축 시에는 호출되었던 스킬이 공유 토큰 예산 한도까지 다시 주입되며, 오래된 스킬부터 먼저 제외됩니다.

모범 사례: 배포 체크리스트, 릴리스 절차, 코드 리뷰 플레이북 같은 절차적 워크플로는 CLAUDE.md가 아니라 스킬에 담으세요.

서브에이전트(Subagents)

서브에이전트는 .claude/agents/에 있는 마크다운 파일로, 특정 곁가지 작업을 위한 독립된 보조 에이전트를 정의합니다. 이름·설명·모델·도구 접근 권한은 YAML 프런트매터로 지정하고, 그 뒤에 시스템 프롬프트 내용이 옵니다.

세션 시작 시에는 메타데이터만 로드되고, Agent 도구로 호출될 때만 본문 전체가 로드됩니다. 서브에이전트는 격리된 컨텍스트 윈도에서 실행되며, 최종 메시지와 메타데이터만 메인 세션으로 돌아옵니다. 최대 5단계까지 중첩할 수 있습니다.

모범 사례: 메인 대화를 어지럽힐 만한 깊은 검색, 로그 분석, 의존성 감사 같은 격리된 곁가지 작업에는 서브에이전트를 쓰세요. 절차가 메인 스레드에서 눈에 보이게 진행되어야 한다면 스킬을 쓰세요.

훅(Hooks)

훅은 사용자가 정의한 명령어, HTTP 엔드포인트, 또는 LLM 프롬프트로, 파일 편집·도구 호출·세션 시작 같은 라이프사이클 이벤트에서 발동합니다. settings.json, 관리형 정책 설정, 또는 스킬/에이전트 프런트매터에 등록합니다.

종류로는 command, HTTP, mcp_tool, prompt, agent 훅이 있습니다. 모두 결정론적으로 발동되며, 설정이 메인 컨텍스트 윈도 밖에 있으므로 컨텍스트 비용이 낮습니다.

모범 사례: 편집 후 린터 실행, 완료 시 Slack 알림, 특정 명령어 실행 차단처럼 결정론적 자동화에 훅을 사용하세요.

출력 스타일(Output Styles)

출력 스타일은 .claude/output-styles/에 있으며 시스템 프롬프트에 지시를 주입합니다. 매 세션 시작 시 로드되고 절대 압축되지 않아, 지시 준수 가중치가 가장 높지만 컨텍스트 비용은 중간 수준입니다.

내장 스타일로는 Proactive, Explanatory, Learning이 있어, 커스텀 스타일 파일을 관리하지 않고도 흔한 요구를 충족합니다. 커스텀 스타일은 기본 지시를 대체하는데, 여기에는 변경 범위 설정, 주석 추가, 보안 처리, 검증 습관에 관한 핵심 지침도 포함됩니다.

모범 사례: 커스텀을 만들기 전에 내장 스타일부터 확인하세요. 기본 시스템 프롬프트를 대체하면서 의도치 않은 동작 변화가 생기지 않도록 주의하세요.

시스템 프롬프트 덧붙이기

append-system-prompt 플래그는 Claude의 핵심 역할을 바꾸지 않고 지시를 추가합니다. 세션 간에 유지되지 않고, 명령 실행 시점에 전달되는 단일 호출에만 적용됩니다.

모범 사례: 특정 코딩 표준, 출력 형식, 도메인 특화 지식에 사용하세요. 지시가 쌓일수록 준수 효과는 점점 줄어듭니다.

맞춤 설정을 위한 빠른 팁

  • CLAUDE.md의 "X 할 때마다 항상 Y 하라": 포매터 실행이나 알림 전송 같은 결정론적 동작에는 훅을 쓰세요.
  • CLAUDE.md의 "이건 절대 하지 마라": 결정론적 가드레일은 프롬프트 지시가 아니라 훅이나 관리형 설정으로 구현해야 합니다.
  • CLAUDE.md에 든 30줄짜리 절차: 호출될 때만 본문이 로드되는 스킬로 옮기세요.
  • 경로 없는 API 전용 룰: 경로 범위를 지정해 무관한 컨텍스트에 룰이 끼어들지 않게 하세요.
  • 프로젝트 파일에 든 개인 취향: 개인 취향은 사용자 레벨 설정에 두고, 프로젝트 파일은 팀 전체 설정용으로 남겨두세요.
