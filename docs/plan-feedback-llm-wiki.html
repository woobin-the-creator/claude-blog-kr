글 평가(좋아요·싫어요·이유) → LLM Wiki 선호 학습

개인 선호 신호를 모아, "내가 선호하는 타입의 정보를 직접 가져오는" 지식 베이스를 만든다.

설계 확정: 2026-06-26 (grill 인터뷰) · 데이터 주인: 나 한 명 · 저장: localStorage + Supabase

목표 한 줄. 글마다 좋아요/싫어요와 그 이유를 남기고, 이 신호를 Karpathy "LLM Wiki" 패턴으로 컴파일해 주제·출처별 취향 프로필을 만든다. 그러면 "내가 좋아할 OO 주제 글 가져와"에 큐레이션으로 답할 수 있다.

0 "LLM Wiki"가 뭔지부터 — 조사한 실제 유즈케이스

막연한 개념이 아니라 Andrej Karpathy가 정의한 패턴이 정확한 출처였다. 핵심: raw 소스를 LLM이 index.md + 위키링크 article로 한 번 컴파일해 누적한다 — 매 질문마다 재검색하는 RAG와 달리 종합(synthesis)을 1급 산출물로 둔다. 우리 규모(글 수십 개)면 벡터DB 없이 index 탐색으로 충분.

Karpathy LLM Wiki

raw → 컴파일 → 쿼리

index를 항상 컨텍스트에 두고 필요한 article만 읽음. 벡터DB 불필요. 우리 블로그 글 = raw 소스.

second-brain (구현 예)

raw/ → wiki/ → output/

YAML frontmatter + wikilink. ingest·query·lint 3워크플로우. 폴더 규약 그대로 차용 가능.

Pistis-RAG · EPIC

선호로 재랭킹

like/dislike 피드백을 RAG에 주입해 결과 순서를 재정렬. 선호 = compact·stable한 개인 컨텍스트.

mem0 · Letta

메모리 레이어

선호 사실 추출→저장→프롬프트 주입. 우린 대화 대신 명시적 평가라 추출이 더 정확.

결정적 정합성: schema.sql의 feedback 컬럼은 이미 "맥 배치(AI)가 채울 자리"로 예약돼 있었다. 즉 원 설계가 사용자 신호 → AI가 읽고 산출물 써넣기 흐름을 의도했다. 사용자 코멘트 = 입력 / feedback = AI 산출, 절대 섞지 않는다.

3 2단계 구현 계획

Phase A — 신호 수집기지금
  • store.js — ratings/reasons 모델 + API + present/ensure/import
  • nav.js — cbk-bar에 👍👎 토글 + 이유칸(평가 시 펼침) + refreshBar 가드
  • library.html — 평가 뱃지·이유·필터칩
  • schema.sql — rating smallint + reason text 컬럼, cbk_upsert/delete 갱신
  • CBK.sync — rating/reason LWW 머지·전달
  • export — 위키용 레코드 포맷(Phase B 입력)
Phase B — LLM Wiki추후
  • wiki/ — index.md · profile.md(선호 프로필) · topics/ · sources/
  • CLAUDE.md — 구조·규약 + ingest/query/lint 워크플로우
  • ingest — export 레코드 → source·topic·profile 갱신 + 위키링크
  • query — index 후보 → profile로 재랭킹 → 큐레이션
  • lint — 모순·끊긴 링크·고아·오래된 주장 점검
  • (선택) 결과를 feedback 컬럼에 써 사이트에 노출

4 핵심 데이터 모델 (Phase A)

기존 cbk:data:v1에 서브객체 2개 추가. notes와 동일 패턴(LWW용 meta 공유). Supabase는 새 컬럼 2개 — feedback 재사용은 금지.

// store.js — localStorage 데이터
{
  bookmarks: { [slug]: true },
  notes:     { [slug]: "string" },
  ratings:   { [slug]: 1 | -1 },     // 신규. 중립이면 키 없음
  reasons:   { [slug]: "string" },   // 신규. 평가 이유
  meta:      { [slug]: "ISO" }       // 모든 신호 공유(per-slug LWW)
}

-- supabase/schema.sql — 새 컬럼 (feedback 은 AI 배치 전용, 건드리지 않음)
alter table public.cbk_items add column if not exists rating smallint;     -- 1 | -1 | NULL
alter table public.cbk_items add column if not exists reason text not null default '';

5 작업 순서

  • store.js — ratings/reasons 모델·API (로컬만, sync 제외)
  • nav.js — cbk-bar 평가 버튼 + 이유칸 + refreshBar 가드
  • library.html — 평가 뱃지·이유·필터
  • schema.sql — 컬럼 2개 + cbk_upsert/cbk_delete 갱신 + grant (Supabase 적용)
  • store.js CBK.sync — rating/reason 머지·전달
  • export — 위키용 레코드 포맷
  • (추후) Phase B 위키 스캐폴드 + CLAUDE.md 워크플로우
