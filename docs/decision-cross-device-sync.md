# 결정 기록 — 기기 간 동기화 방식

- **상태:** 채택됨 (현재: 단일 사용자, 비밀 `sync_key` 방식)
- **날짜:** 2026-06-28
- **관련 코드:** `posts/assets/store.js`(CBK.sync), `posts/assets/cbk-config.js`, `supabase/schema.sql`, `library.html`

## 맥락

- 사이트는 정적 호스팅(GitHub Pages, `woobin-the-creator/claude-blog-kr`).
- 즐겨찾기·메모·평가(좋아요/싫어요·이유)는 **개인 선호 로그**로, 현재 사용자는 사실상 **본인 1명**.
- 저장은 Supabase `cbk_items` 한 줄/글. 테이블은 RLS 전면 차단, 브라우저는 `cbk_get`/`cbk_upsert`/`cbk_delete` RPC(anon 키)로만 접근. 데이터를 지키는 비밀은 anon 키가 아니라 **24자 무작위 `sync_key`**.

## 결정

기기 간 동기화는 **비밀 `sync_key`(passphrase) 방식을 유지**한다. **구글 로그인/Supabase Auth는 도입하지 않는다.**

이유: 현재 단일 사용자(A 시나리오)에서는 `sync_key` 코드 하나가 이미 "내 여러 기기" 동기화를 해결한다. 구글 로그인을 붙이면 OAuth 클라이언트 발급·RLS 재설계·supabase-js 의존성이 새로 필요한데, 그 대가로 얻는 건 "코드 대신 구글 계정"이라는 친숙함뿐이고 그 수혜자도 본인 1명이다. 비용 대비 이득이 없다.

## 잠재 페인 포인트 — 공개/다중 사용자로 전환할 때 (재검토 트리거)

**트리거:** 사이트를 공개해 **방문자가 각자 자기 즐겨찾기·평가를 보관**하게 만들 때.

이 순간 `sync_key` 모델은 깨진다 — 공유 비밀 하나로는 사용자를 분리할 수 없고, 코드를 아는 사람은 **모두의 데이터**에 접근하게 된다. 그때는 사용자별 인증으로 전환해야 한다:

1. **Supabase Auth** 도입(구글 OAuth 또는 매직링크) + `supabase-js`를 사이트에 추가.
2. 데이터 식별 키를 `sync_key` → **`auth.uid()`**(로그인 계정 ID)로 교체. `cbk_items`에 `user_id` 추가, **RLS를 `auth.uid()` 기준**으로 재설계(각자 자기 행만).
3. **Google Cloud OAuth 클라이언트** 발급 + Pages 주소를 **승인된 리다이렉트 URI**로 등록.
4. 기존 `sync_key` 데이터의 **마이그레이션 경로** 고려(또는 폐기).

**주의:** 로그인으로 바꿔도 "기기당 1회 인증"이라는 단계는 사라지지 않는다. 로그인의 값어치는 *다중 사용자 분리*이지 *절차 제거*가 아니다. 동적 서버가 필요한 것도 아니다(정적 사이트 + Supabase Auth로 가능).

## 별개의 개선 (지금도 가능, 로그인 무관)

수동 "지금 동기화" 버튼의 불편은 **자동 동기화**(페이지 로드 시 자동 pull, 변경 시 디바운스 자동 push)로 없앨 수 있다. 이는 정적 사이트 + 현재 `sync_key` 구조 그대로 구현 가능하며, 서버도 로그인도 필요 없다.
